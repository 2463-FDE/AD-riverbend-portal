# W10 — Append-only audit trail (CAPSTONE)

> Per-engagement spec for curriculum Week 10. Client ask + decoded defect fixed by the program;
> scope / deliverables / acceptance criteria are ours. Capstone — integrates W1–W9 findings.

- **Status:** derived by `/dashboard` from live repo state — not stored here (see §5).
- **Debt IDs:** D2 (mutable "audit" log, not tamper-evident), D12 (disclosure accounting)
- **Tickets:** none (COO "prove we're audit-ready, package it for the board")
- **Deliverable type:** capstone slice — build (audit trail + alert) + report/package (board-facing)

## 1. Client ask (verbatim)
> Home stretch — I need to prove to the board that we're compliant and audit-ready, and package
> everything up. The auditor sent a pre-visit questionnaire and a couple of the questions completely
> stumped my team. Can you pull the whole engagement together into something I can present? I want it
> to land in terms the board actually cares about.

## 2. What it really is
- Capstone: an **append-only access trail** (replacing the mutable `audit_logs`) + a
  **breach-detection alert** + an **accounting-of-disclosures report generator** + a
  **debt register / ADR catalog / risk-prioritized roadmap**, framed for a board audience
  (audit-readiness, patient-safety, financial). The stumped auditor questions are in
  `docs/handover/auditor-questionnaire.md`.

## 3. Scope
### In scope
- Append-only / tamper-evident access trail replacing mutable `audit_logs`.
- Breach-detection alert (e.g. anomalous cross-patient access).
- Accounting-of-disclosures report generator (feeds ROI / 164.528).
- Board-facing package: debt register, ADR catalog, risk-prioritized roadmap.
- Answers to the stumped auditor-questionnaire questions.
### Explicitly OUT
- Re-doing earlier weeks' impl — capstone *integrates* W1–W9, it's a slice not full re-scope.

## 4. Deliverables
1. Append-only access trail (schema + write path).
2. Breach-detection alert.
3. Accounting-of-disclosures report generator.
4. Board package: debt register + ADR catalog + risk-prioritized roadmap.
5. Auditor-questionnaire answers.

## 5. Acceptance criteria (done-definition)
- [ ] Access trail is append-only / tamper-evident — a test proves an update/delete is rejected or detectable.
- [ ] Every finalized chart view / disclosure writes an access event ("who viewed patient Y").
- [ ] Breach-detection alert fires on a demonstrated anomalous-access scenario.
- [ ] Disclosure-accounting report generates a 164.528-style record for a patient.
- [ ] Board package maps each debt (D1–D14) to status + residual risk + roadmap priority.
- [ ] The stumped auditor-questionnaire questions each have a written answer.

## 6. Landmines (CLAUDE.md §6/§9)
- ⚠️ Mutable "audit" log is not tamper-evident (D2/D12) — replacing it touches the compliance boundary.
- ⚠️ ROI / disclosure logic touches PHI + compliance — human approval before behavior changes.
- ⚠️ Schema change → update both `db/schema.sql` and a new `db/migrations/00N_*.sql` (hand-synced).
