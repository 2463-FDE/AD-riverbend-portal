## 1. Client ask (verbatim)
> Our scheduling keeps double-booking the same slot and it's getting embarrassing — two patients
> show up for one appointment. The team's pretty sure it's a glitch in the calendar widget, so could
> you just fix the calendar so it stops doing that? … It seems to happen more when the system's busy.

## 2. What it really is
- **NOT a UI bug.** "Glitch in the calendar widget" is the client's wrong mental model. It's a
  **check-then-insert concurrency defect** in `scheduling-service/book.py` — no UNIQUE on `slot_id`,
  no idempotency key. "More when busy" **confirms a race**, it doesn't point at the UI.

## 3. Scope
### In scope
- Reframe the problem from "calendar glitch" to "check-then-insert race / missing idempotency."
- Acceptance criteria for a correct fix.
- Test vectors: concurrent-booking and retry/duplicate-submit scenarios.
### Explicitly OUT
- **Any implementation.** Do not add the UNIQUE constraint, idempotency key, or locking this week.
- Frontend calendar changes (the wrong fix).

## 4. Deliverables
1. Spec doc: problem reframe (why it's a race, not a widget bug).
2. Acceptance criteria for the eventual fix.
3. Test vectors (concurrent double-book; duplicate confirmation on retry).

## 5. Acceptance criteria (done-definition — for THIS week's spec artifact)
- [ ] Spec explicitly refutes the "calendar widget" theory with the concurrency evidence.
