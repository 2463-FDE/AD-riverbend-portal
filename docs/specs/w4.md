- **Status:** derived by `/dashboard` from live repo state — not stored here (see §5).
- **Debt IDs:** D11 (IDOR), D8 (N+1 / full-table scan)

## 1. Client ask (verbatim)

Persona: Dr. Maya Okonkwo, COO, Riverbend Community Health.

> Patients keep calling to ask how to see their own labs and visit summaries — right now it's
> all scattered across services. Can you build something that assembles a patient's full picture
> in one view? The portal already has a records endpoint you can build on. Make it feel fast,
> some of the record pages are a bit sluggish today. I'll send the network capture our QA person
> grabbed off the current portal.

## 2. What it really is (decoded defect)

- **"records endpoint you can build on"** is the IDOR surface itself: `GET /patients/{id}/records`
  requires a session but never binds it to `{patient_id}`, and IDs are sequential/walkable. Building
  on it as-is **propagates the bug** (D11).
- **"sluggish record pages"** = N+1 / unindexed reads in the records read + search paths (D8).
- The **network capture** (`docs/handover/portal.har`) is expected to reproduce the IDOR (one
  session reading a patient it does not own).

## 3. Scope

### In scope
- IDOR analysis of `GET /patients/{id}/records` (and sibling record reads) — how the missing
  ownership/care-relationship check lets a session walk sequential IDs.
- N+1 / full-scan analysis of the records read + search paths (D8).
- A **multi-agent retrieval prototype** that assembles a patient view (orchestrator + retriever(s)
  + assembler) — prototype quality, to demonstrate the assembly shape, not a production feature.
- A **KG / patient-view schema** sketch (encounters · records) the assembly reads from.
- The **IDOR writeup** + HAR repro (primary deliverable).

### Explicitly OUT of scope (defer to the owning week)
> This is the section that keeps a "north-star" diagram from being mistaken for the W4 deliverable.
- **D7 minimum-necessary / RBAC** → W9. Do not build role/scope enforcement here.
- **D1 LLM grounding guard** → W1 (shipped) / W7. Do not add a grounding gate as W4 scope.
- **D13 PHI-to-vendor / redaction-before-boundary** → W8.
- **D2 / D12 append-only audit trail + disclosure accounting** → W10 capstone.
- **HITL gates** (break-glass, clinician confirm) — not in any current spec; note as a future
  question, do not build.
- **D10 session TTL** — borderline (tagged W4/W9 in CLAUDE.md §9). Analyze/flag if it intersects
  the IDOR fix, but do not ship session-expiry behavior under this week without human approval.

## 4. Deliverables (artifacts that must exist for "Done")

1. `docs/…` IDOR writeup — the vuln, the HAR repro steps, the fix (bind session identity →
   `patient_id` via ownership or care-relationship check), and why the fix touches the auth boundary.
2. Multi-agent retrieval **prototype** (code, clearly marked prototype).
3. KG / patient-view **schema sketch** (doc or `.sql` proposal — not a shipped migration unless approved).
4. D8 N+1 / full-scan finding (batched fetch + indexed search recommendation, with evidence).
5. ADR if a non-trivial design decision is made (per project rule: decisions get an ADR, not just code).

## 5. Acceptance criteria (done-definition — the diffable part)

- [ ] IDOR is **demonstrated**: a characterization test (or documented HAR repro) shows a session
      reading a patient it does not own **succeeds today** (matches the existing `xfail`).
- [ ] Proposed fix **binds session identity to `{patient_id}`**; writeup states whether the bind is
      patient-ownership or care-relationship, and names the missing table if one is required.
- [ ] D8: at least one read path shown to issue N+1 / full-scan queries, with a bounded/indexed
      alternative proposed (evidence, not assertion).
- [ ] Prototype assembles a patient view **through the gateway**, never calling a domain service
      directly (portal → gateway → service invariant, CLAUDE.md §1).
- [ ] No out-of-scope debt (D7/D1/D13/D2/D12) is silently implemented; anything touched there is
      flagged, not shipped.
- [ ] Any auth-boundary change is **flagged for human approval** and NOT merged without it
      (CLAUDE.md §6). W4's shippable output is analysis + prototype; the live IDOR fix may land later.
- [ ] Adversarial / negative test for the authz check exists if any authz code is written
      (RIV / PR #2 negative-test rule, CLAUDE.md §5).

## 6. Relevant landmines (CLAUDE.md §6 / §9)

- ⚠️ **IDOR on chart reads** — `GET /patients/{id}/records` requires a session but never binds it to
  `{patient_id}`; IDs sequential + walkable. Intentional, documented-in-code gap. **Never change auth
  behavior without explicit human approval.**
- ⚠️ **N+1 / full-table scans** in records read/search paths (D8/D10 amplified · W4).
- ⚠️ **PHI handling** — `notes`, `ssn` plaintext; do not widen exposure when assembling a "full view."
- ⚠️ **IDOR prevention is an `xfail`** in the suite (cross-patient reads currently succeed) — do not
  "fix" the test to hide the gap; characterize first.

## 7. Open questions (resolve before this becomes an ADR)

1. Does "session bind" mean patient-ownership or a care-relationship check? Riverbend has **no such
   table yet** — a care-relationship model may itself be a schema deliverable.
2. Is the multi-agent fan-out worth it vs. a single retriever, given the real deliverable is the
   IDOR writeup + HAR repro?
3. Which (if any) HITL gates are mandatory vs. break-glass-only? (Latency cost — likely defer to W9/W10.)
