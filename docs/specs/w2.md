# W2 — RAG dedup / fragmentation eval

> Per-engagement spec for curriculum Week 2. Client ask + decoded defect fixed by the program;
> scope / deliverables / acceptance criteria are ours.

- **Status:** derived by `/dashboard` from live repo state — not stored here (see §5).
- **Debt IDs:** D5a (duplicate patients, no MPI)
- **Tickets:** RIV-160 (allergy differs per chart — dup-chart facet)
- **Deliverable type:** build (eval harness) + ADR

## 1. Client ask (verbatim)
> Our clinicians are constantly complaining that they have to click through a patient's entire
> history to find anything relevant. Could you build a retrieval helper that surfaces the most
> relevant past records the moment they open a chart? The last contractor left a little 'gold set'
> … it all looked great when he demoed it. … Don't go too crazy on cost, our AI budget is basically
> a Claude Pro plan.

## 2. What it really is
- The "gold set that looked great" is **suspect** — likely overfit/rigged to demo well. The real
  finding is that the same patient exists as **duplicate, fragmented charts** (no MPI/match key), so
  retrieval returns a partial history. Deliverable is a **RAG eval harness** that measures dup-rate /
  fragmentation, plus an **MPI / match-key ADR**.

## 3. Scope
### In scope
- Eval harness measuring retrieval quality: duplicate-patient rate, chart fragmentation, and whether
  the gold set is representative vs. overfit.
- Fragmentation report over the exported patients/encounters data.
- MPI / match-key **ADR** (how Riverbend should identity-resolve patients).
### Explicitly OUT (owning week)
- Implementing the MPI / merging charts — ADR-level only this week; impl is future/separate.
- HL7 AL1/RXA drop (the *other* half of RIV-160) → W6.
- Cost/model-provider hardening → W1/observability track.

## 4. Deliverables
1. RAG eval harness (runnable; dup-rate + fragmentation metrics).
2. Fragmentation report (findings over real exported data).
3. MPI / match-key ADR.
4. Characterization tests for the eval logic.

## 5. Acceptance criteria (done-definition)
- [ ] Harness computes duplicate-patient rate and fragmentation on the provided export, with numbers.
- [ ] Harness demonstrates the gold set is **not** representative (or proves it is) — not asserted, shown.
- [ ] Report names concrete duplicate patients (e.g. the RIV-160 "Maria Gonzalez" facet) as evidence.
- [ ] MPI ADR states the proposed match key and the trade-offs (false-merge vs. false-split).
- [ ] Similarity/clustering logic is tested against the input-trust bug class (n≥3 chains, ambiguous
      bucket) — not just a happy-path pair (fix-the-class lesson, PR #4 r1→r4).

## 6. Landmines (CLAUDE.md §6/§9)
- ⚠️ Duplicate patients — self-service intake has no MPI/match key (RIV-160).
- ⚠️ Plaintext PHI in the exported tables — keep it inside the eval boundary.

## 7. Open questions
1. Match key basis — deterministic (name+DOB+…) vs. probabilistic? ADR must pick and defend.
2. Is a false-merge (two people fused) or a false-split (one person duplicated) the worse failure here?
