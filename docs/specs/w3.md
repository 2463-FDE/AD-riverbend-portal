# W3 — Eligibility agent (timeout / circuit-breaker)

> Per-engagement spec for curriculum Week 3. Client ask + decoded defect fixed by the program;
> scope / deliverables / acceptance criteria are ours.

- **Status:** derived by `/dashboard` from live repo state — not stored here (see §5).
- **Debt IDs:** D4 (synchronous payer call, no timeout/breaker)
- **Tickets:** RIV-088 (registration spins ~4-5s), RIV-141 (whole intake froze Tue 9:00–9:20)
- **Deliverable type:** build + ADR

## 1. Client ask (verbatim)
> Front desk is miserable doing insurance eligibility checks … Can you build them a little chat
> assistant that checks a patient's eligibility and keeps track of the visit context as they go?
> Heads up: eligibility 'gets slow' now and then, but it usually sorts itself out. … one of the
> front-desk leads filed a ticket about a rough morning last week; I'll forward it, probably nothing.

## 2. What it really is
- "Gets slow, sorts itself out" + "rough morning" = **RIV-088/RIV-141**: the inline eligibility call
  is synchronous with no timeout, so a payer outage **freezes intake entirely** (not just eligibility).
  Real deliverable: eligibility agent with **timeout + circuit-breaker** + **visit-scoped memory** +
  an ADR (sync→async, graceful degradation).

## 3. Scope
### In scope
- Bound the payer call (per-request timeout) and add a circuit-breaker so an outage degrades
  gracefully instead of freezing intake.
- Decouple intake from eligibility (payer slowness must not block registration Save).
- Visit-scoped conversation memory for the agent.
- ADR: sync→async direction + cache/degradation strategy.
### Explicitly OUT (owning week)
- Rewriting the payer X12 270/271 shim itself.
- IDOR / records assembly → W4.
- Full async job infrastructure — ADR may *propose* it; this week bounds + breaks the call.

## 4. Deliverables
1. Eligibility agent with timeout + circuit-breaker.
2. Intake decoupled from the payer call path.
3. Visit-scoped memory.
4. ADR (sync→async, graceful degradation).
5. Regression test proving intake no longer freezes on payer outage.

## 5. Acceptance criteria (done-definition)
- [ ] Payer call has a hard per-request timeout; proven by test.
- [ ] Circuit-breaker opens on repeated failure and is **thread-safe**; proven by test.
- [ ] Payer status classification is correct (outage ≠ "not eligible"); adversarial test exists.
- [ ] Intake Save does **not** block on payer slowness — regression test proves the RIV-141 freeze is gone.
- [ ] Payer/intake timeout budgets are aligned (intake budget ≥ payer timeout).
- [ ] ADR documents the sync→async decision and the degradation contract.

## 6. Landmines (CLAUDE.md §6)
- ⚠️ Inline eligibility call (`intake` → `eligibility`) is synchronous, no timeout — a payer outage
  freezes intake. This is the exact defect; the fix must preserve the PHI boundary on unexpected errors.
- ⚠️ Proxy helpers swallow errors into `{"error": …}` 200 OK — don't let the breaker misclassify.

## 7. Open questions
1. Does eligibility get cached, and for how long, when the breaker is open (stale-but-usable vs. fail)?
2. Is visit-scoped memory in Redis (session-adjacent) or agent-local?
